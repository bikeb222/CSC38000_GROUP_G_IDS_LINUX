# System Architecture

## Overview

The system runs inside Ubuntu on WSL. Suricata monitors traffic, compares it against signature rules, and writes alerts to log files. EveBox reads the Suricata EVE JSON log and displays alerts through a web GUI.

```text
Windows Host
   |
   | WSL Ubuntu
   |
   |-- Suricata IDS
   |     |-- ET Open ruleset
   |     |-- Custom local.rules
   |     |-- /var/log/suricata/fast.log
   |     |-- /var/log/suricata/eve.json
   |
   |-- EveBox GUI
         |-- Reads /var/log/suricata/eve.json
         |-- Displays alerts in a browser dashboard
```

## Main Components

Suricata IDS is the detection engine. It inspects packets and application-layer traffic, evaluates signature rules, and creates alerts when rules match.

ET Open rules provide community-maintained threat detection signatures. They are installed and updated with `suricata-update`.

Custom `local.rules` provides project-specific signatures for controlled testing. These rules are easy to demonstrate because they match predictable local traffic.

EVE JSON is Suricata's structured event log format. EveBox uses this log as its alert data source.

EveBox is the GUI layer. It does not perform detection; it helps the analyst review alerts generated by Suricata.

## Data Flow

1. Test traffic is generated inside Ubuntu WSL.
2. Suricata captures traffic from an interface such as `eth0` or `lo`, or reads packets from a PCAP file.
3. Suricata compares the traffic with ET Open rules and custom local rules.
4. Matching traffic produces alerts in `fast.log` and `eve.json`.
5. EveBox reads `eve.json` and displays alert details in the browser.

## WSL Considerations

WSL networking may not mirror a full Linux VM. Some traffic may not be visible on the expected interface. The project therefore includes two validation paths: live monitoring and offline PCAP analysis.
